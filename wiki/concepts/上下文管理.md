# 上下文管理

LLM 上下文窗口的优化策略，解决工具和 MCP 配置占用过多 Token 的问题。

## 核心问题

LLM 的上下文窗口有限（如 Claude 200k tokens），但工具定义、MCP 配置、系统提示会占用大量 Token：

- **原始窗口**：200k tokens
- **被占用后**：可能只剩 70k tokens
- **影响**：可用上下文大幅缩减，影响复杂任务处理

## Everything Claude Code 的警告

> 启用太多 MCP 会将 200k 上下文窗口压缩到 70k。

经验法则：
- 配置 20-30 个 MCP（全局）
- 每个项目保持启用少于 10 个
- 活动工具少于 80 个

## 优化策略

### MCP 选择性禁用

使用 `disabledMcpServers` 在项目配置中禁用未使用的服务：

```json
{
  "disabledMcpServers": ["github", "context7", "exa"]
}
```

### 系统提示精简

- 使用最小化的系统提示
- 按需加载技能而非全局加载
- 使用 contexts/ 目录按模式注入上下文

### 后台进程

- 将低优先级任务移至后台智能体
- 使用 subagent 减少主上下文占用

### 策略性精简

Everything Claude Code 的 `strategic-compact` 技能：
- 分析上下文占用
- 提出精简建议
- 保留关键信息

## 与 oh-my-openagent 的对比

| 框架 | 上下文管理策略 |
|------|---------------|
| ECC | disabledMcpServers + strategic-compact |
| oh-my-openagent | 按需 MCP（任务完成即销毁） |

两者都意识到 MCP 全局加载的问题，但解决方案不同。
